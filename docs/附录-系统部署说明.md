# 附录：W4Agent 系统部署说明

本文档用于说明“基于智能体技术的 Web 无障碍检测系统 W4Agent”的部署方法。系统采用前后端分离架构，前端基于 React 与 Ant Design 构建，后端基于 FastAPI 提供 REST API 与 WebSocket 服务，并依赖 PostgreSQL、Redis、Playwright 浏览器环境以及大语言模型接口完成网页遍历、无障碍检测与报告生成。为保证部署过程可复现，本文优先介绍 Docker Compose 一键部署方式，同时给出本地开发部署方式作为补充。

## 一、部署环境要求

系统部署前需要准备如下软硬件环境。

### （一）硬件环境

| 项目 | 建议配置 |
| --- | --- |
| CPU | 4 核及以上 |
| 内存 | 8 GB 及以上 |
| 磁盘 | 20 GB 及以上可用空间 |
| 网络 | 能够访问待检测网站及所配置的大语言模型接口 |

### （二）软件环境

| 软件 | 版本要求 | 用途 |
| --- | --- | --- |
| Docker | 24.0 及以上 | 构建和运行容器 |
| Docker Compose | v2 及以上 | 编排前端、后端、数据库和缓存服务 |
| Git | 2.0 及以上 | 获取项目代码 |

若采用本地开发方式部署，还需要额外安装：

| 软件 | 版本要求 | 用途 |
| --- | --- | --- |
| Python | 3.11 及以上 | 运行后端服务 |
| Node.js | 20 及以上 | 构建和运行前端服务 |
| PostgreSQL | 16，需支持 pgvector | 存储任务、页面、问题和记忆数据 |
| Redis | 7 及以上 | 缓存和实时任务状态辅助存储 |

## 二、系统组成

部署后的系统由以下服务组成：

| 服务名称 | 默认端口 | 说明 |
| --- | --- | --- |
| frontend | 5173 | 前端页面服务，基于 Nginx 托管构建后的静态资源 |
| backend | 8000 | 后端 API 服务，提供任务管理、检测执行、报告导出和 WebSocket 推送 |
| postgres | 5432 | PostgreSQL 数据库，使用 pgvector 镜像支持向量记忆数据 |
| redis | 6379 | Redis 缓存服务 |

其中，前端通过 `/api/v1` 访问后端 REST 接口，通过 `/ws` 建立 WebSocket 连接接收任务执行进度。Docker 部署时，前端 Nginx 已内置 API 与 WebSocket 反向代理配置，用户只需访问前端端口即可使用系统。

## 三、Docker Compose 部署方式

### （一）获取项目代码

在服务器或本机终端中执行以下命令，将代码克隆到部署目录：

```bash
git clone <项目仓库地址> W4Agent
cd W4Agent
```

若代码已通过压缩包或其他方式复制到部署机器，则直接进入项目根目录即可。项目根目录下应包含 `docker-compose.yml`、`backend/`、`frontend/`、`README.md` 等文件或目录。

### （二）配置环境变量

项目根目录提供了 `.env.example` 文件，可复制为实际运行配置文件：

```bash
cp .env.example .env
```

然后编辑 `.env` 文件，根据实际环境修改以下关键配置：

```env
APP_ENV=development
DEBUG=true

LLM_PROVIDER=openai
OPENAI_API_KEY=your-api-key
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4o

CORS_ORIGINS=["http://localhost:5173"]
SECRET_KEY=please-change-this-secret-key
```

主要配置项说明如下：

| 配置项 | 说明 |
| --- | --- |
| APP_ENV | 应用运行环境，可设置为 development、staging 或 production |
| DEBUG | 是否开启调试模式，生产环境建议设置为 false |
| LLM_PROVIDER | 大语言模型提供方，可选 openai、claude 或 local |
| OPENAI_API_KEY / ANTHROPIC_API_KEY | 对应模型服务的访问密钥 |
| OPENAI_BASE_URL / ANTHROPIC_BASE_URL | 模型服务接口地址，可替换为兼容接口或代理地址 |
| OPENAI_MODEL / ANTHROPIC_MODEL | 实际使用的模型名称 |
| LOCAL_LLM_BASE_URL | 使用本地模型时的服务地址，例如 Ollama 或 vLLM 地址 |
| BROWSER_POOL_SIZE | Playwright 浏览器池大小，影响并发检测能力和资源占用 |
| PAGE_TIMEOUT_MS | 页面加载和操作超时时间，单位为毫秒 |
| CORS_ORIGINS | 允许访问后端接口的前端地址列表 |
| SECRET_KEY | JWT 加密密钥，生产环境必须替换为随机强密钥 |

Docker Compose 中已经为后端容器设置了容器内数据库和 Redis 地址，因此通常不需要在 `.env` 中修改 `DATABASE_URL` 与 `REDIS_URL`。若需要连接外部数据库或外部 Redis，可同步修改 `docker-compose.yml` 中 backend 服务的环境变量。

### （三）构建并启动服务

在项目根目录执行：

```bash
docker compose up -d --build
```

该命令会完成以下操作：

1. 拉取 PostgreSQL pgvector 镜像与 Redis 镜像；
2. 构建后端镜像，安装 Python 依赖、Playwright Chromium 浏览器和中文字体；
3. 构建前端镜像，执行 TypeScript 编译与 Vite 打包；
4. 启动数据库、缓存、后端和前端服务；
5. 后端容器启动时自动执行 Alembic 数据库迁移。

启动完成后，可执行以下命令查看容器状态：

```bash
docker compose ps
```

若所有服务状态均为 `running` 或 `healthy`，表示系统已正常启动。

### （四）访问系统

默认部署完成后，在浏览器中访问：

```text
http://localhost:5173
```

如果部署在远程服务器上，则将 `localhost` 替换为服务器 IP 地址或域名：

```text
http://<服务器IP或域名>:5173
```

后端 API 文档可通过以下地址访问：

```text
http://localhost:8000/docs
```

首次进入系统后，可在前端页面中新建项目和检测任务，填写待检测网页 URL、检测深度、WCAG 等级等参数，然后启动检测。任务执行过程中，前端会通过 WebSocket 实时显示页面遍历、问题发现、智能体推理日志和报告生成进度。

## 四、本地开发部署方式

本地开发部署适用于调试代码、修改功能或单独运行前后端服务的场景。

### （一）启动数据库和 Redis

可以只使用 Docker 启动 PostgreSQL 与 Redis：

```bash
docker compose up -d postgres redis
```

此时数据库地址为：

```text
postgresql+asyncpg://w4agent:w4agent_pass@localhost:5432/w4agent
```

Redis 地址为：

```text
redis://localhost:6379/0
```

### （二）安装后端依赖

进入后端目录并安装 Python 依赖：

```bash
cd backend
pip install -e ".[dev]"
playwright install chromium
```

随后执行数据库迁移：

```bash
alembic upgrade head
```

启动后端服务：

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

后端启动成功后，可访问 `http://localhost:8000/docs` 查看接口文档。

### （三）安装前端依赖

新开一个终端，进入前端目录：

```bash
cd frontend
npm install
npm run dev
```

前端开发服务器默认监听 `5173` 端口，访问地址为：

```text
http://localhost:5173
```

开发模式下，Vite 会将 `/api`、`/ws` 和 `/static` 请求代理到 `http://localhost:8000`，因此前端无需额外配置后端地址。

## 五、常用运维命令

项目根目录的 `Makefile` 对常用命令进行了封装，也可以直接使用 Docker Compose 命令。

| 操作 | Make 命令 | Docker Compose 命令 |
| --- | --- | --- |
| 启动全部服务 | `make up` | `docker compose up -d` |
| 停止全部服务 | `make down` | `docker compose down` |
| 构建镜像 | `make build` | `docker compose build` |
| 查看运行日志 | `make logs` | `docker compose logs -f` |
| 查看服务状态 | - | `docker compose ps` |
| 执行数据库迁移 | `make migrate` | `docker compose exec backend alembic upgrade head` |

如需查看某个服务日志，可执行：

```bash
docker compose logs -f backend
docker compose logs -f frontend
docker compose logs -f postgres
docker compose logs -f redis
```

## 六、数据持久化与文件存储

Docker Compose 部署时，系统使用以下持久化卷和挂载目录：

| 名称 | 类型 | 说明 |
| --- | --- | --- |
| pgdata | Docker volume | 保存 PostgreSQL 数据 |
| redisdata | Docker volume | 保存 Redis 数据 |
| reports | Docker volume | 保存导出的检测报告 |
| ./data/screenshots | 本地目录挂载 | 保存检测过程中的网页截图 |

停止服务时执行 `docker compose down` 不会删除数据库和报告数据。若执行 `docker compose down -v`，则会删除 Docker volume，数据库、缓存和报告数据也会被清空，因此正式部署环境中应谨慎使用。

## 七、生产环境部署建议

如果系统部署在公网或学校服务器上，建议进行如下配置：

1. 将 `APP_ENV` 设置为 `production`，并将 `DEBUG` 设置为 `false`；
2. 修改 `SECRET_KEY` 为随机生成的强密钥，避免使用示例值；
3. 将 `.env` 文件加入保密管理，不上传到公开代码仓库；
4. 根据实际域名修改 `CORS_ORIGINS`，例如 `["https://your-domain.example"]`；
5. 使用 Nginx、Caddy 或学校统一网关在前端服务外层配置 HTTPS；
6. 不建议在公网直接暴露 PostgreSQL 的 `5432` 端口和 Redis 的 `6379` 端口；
7. 定期备份 PostgreSQL 数据卷、报告文件和截图文件；
8. 根据机器性能调整 `BROWSER_POOL_SIZE`，避免 Playwright 浏览器实例过多导致内存不足。

一个典型的公网访问结构如下：

```text
浏览器
  │
  ▼
HTTPS 反向代理（Nginx / Caddy / 学校网关）
  │
  ▼
frontend:80
  │
  ├── /api/v1  → backend:8000
  └── /ws      → backend:8000
```

## 八、部署验证流程

完成部署后，可按照以下步骤验证系统是否正常运行：

1. 执行 `docker compose ps`，确认 `postgres`、`redis`、`backend` 和 `frontend` 均已启动；
2. 访问 `http://localhost:5173`，确认前端页面能够正常打开；
3. 访问 `http://localhost:8000/docs`，确认后端 API 文档能够正常打开；
4. 在前端新建一个项目；
5. 新建检测任务，输入一个可访问的测试网页 URL；
6. 启动检测任务，观察任务详情页中的实时进度和日志；
7. 检测完成后进入报告页面，确认问题统计、页面列表、截图和导出功能正常。

若上述步骤均可完成，则说明系统部署成功。

## 九、常见问题及处理方法

### （一）前端页面能打开，但请求接口失败

首先检查后端容器是否正常运行：

```bash
docker compose logs -f backend
```

如果后端日志中出现数据库连接失败，应检查 PostgreSQL 服务状态和 `DATABASE_URL` 配置。如果出现跨域错误，应检查 `.env` 中的 `CORS_ORIGINS` 是否包含当前前端访问地址。

### （二）后端启动时数据库迁移失败

可手动执行迁移命令查看详细错误：

```bash
docker compose exec backend alembic upgrade head
```

常见原因包括 PostgreSQL 尚未完全启动、数据库账号密码配置不一致，或数据库 volume 中存在旧版本表结构。若是测试环境且不需要保留数据，可清理 volume 后重新启动；正式环境应先备份数据再处理。

### （三）检测任务启动后浏览器相关报错

本系统依赖 Playwright Chromium 访问并分析目标网页。Docker 部署时浏览器和依赖库已在后端镜像中安装。若采用本地开发部署，需要确认已执行：

```bash
playwright install chromium
```

如果服务器内存较小，可适当降低 `.env` 中的 `BROWSER_POOL_SIZE`。

### （四）大语言模型调用失败

应检查以下内容：

1. `LLM_PROVIDER` 是否与实际配置一致；
2. API Key 是否有效；
3. `OPENAI_BASE_URL` 或 `ANTHROPIC_BASE_URL` 是否可访问；
4. 模型名称是否被当前接口支持；
5. 服务器网络是否允许访问模型服务。

如果只是验证系统基础功能，可先配置可用的兼容模型接口，待系统部署成功后再切换为正式模型。

### （五）报告或截图无法访问

Docker 部署时截图目录挂载到 `./data/screenshots`，报告保存到 `reports` volume。若页面中无法显示截图，应检查后端日志以及该目录是否具有读写权限。重新构建或迁移部署时，应同时保留数据库数据、截图目录和报告 volume，否则历史任务中的截图或报告可能无法继续访问。

## 十、停止与卸载

停止系统但保留数据：

```bash
docker compose down
```

停止系统并删除数据库、缓存和报告 volume：

```bash
docker compose down -v
```

删除全部容器和镜像前，应确认重要检测数据、截图和报告已完成备份。
